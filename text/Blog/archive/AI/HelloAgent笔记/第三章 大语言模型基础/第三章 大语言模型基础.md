# Transformer

## token

先将一句话拆成一个个最小语义单元, 就得到了一个个token. 现在常用的分词工具（如 BPE、WordPiece）会自动处理这件事，并且会维护一个巨大的**词汇表（Vocabulary）** ，里面存了所有可能出现的 Token。每个 Token 都有一个唯一的 ID（比如 `我=101`​，`爱=102`）。

有了 Token ID 之后，我们通过一个 ​**Embedding 层**（本质是一个巨大的 “查找表”）把它变成向量。

- 假设我们设定向量维度是 512（这是 Transformer 原文的默认值），词汇表大小是 3 万。
- 那么 Embedding 层就是一个形状为 `[30000, 512]` 的矩阵。
- 当我们输入 Token ID（比如 `101`），就直接从这个矩阵里取出第 101 行，这就是这个 Token 的初始向量。

**关键点**：这个 Embedding 矩阵不是人为设定的，而是**模型参数**的一部分。它会在训练过程中，根据任务目标（比如翻译、填空）自动调整，让意思相近的 Token 向量在空间中离得更近。

## attension

token本身是一个向量或者说embedding, 与三个矩阵相乘分别生成了这个token的QKV向量. 对整个上下文的token都生成了这三个向量, 然后对于其中某个token, 通过attension计算公式(用这个tokne的Q来查询每个tokne的K, 以及一些其他计算), 就能得到这个token在这个上下文中的新向量(新位置?)

QKV 矩阵是怎么得到的？

简单来说：**WQ,WK,WV 这三个矩阵，也是模型 “学” 出来的，它们就是可训练的参数。**

1. 初始化：随机开始

在训练开始前，这三个矩阵会被​**随机初始化**（比如用 Xavier 或 He 初始化方法）。

- 假设输入的 Token 向量维度是 dmodel​\=512，我们希望生成的 Q、K、V 向量维度是 dk​\=64。
- 那么 WQ​,WK​,WV​ 的形状就都是 `[512, 64]`。

2. 训练：自动调整

在训练过程中，模型会根据最终的损失（Loss），通过**反向传播（Backpropagation）**  和**梯度下降（Gradient Descent）**  自动更新这三个矩阵里的数值。

- 直观理解：模型会慢慢学会 “什么样的 Q 去查什么样的 K，能最好地提取上下文信息”。
- 比如，在翻译任务中，模型可能会学到让代词（如 “它”）的 Q 去强烈匹配前文某个名词（如 “猫”）的 K。

## 多头注意力

假设单头情况下, 也就是上面写的attension部分的计算方式, 计算出512维的QKV向量. 然后再进一步计算attension(512)

那么多头的情况下, 依旧是乘上QKV矩阵, 然后得到一个512维的QKV, 只不过这时会将QKV按维度分组(Q1 = Q的1-8维, Q2 = Q的9-16维这样类推). 假设8头, 就是每组512/8 = 64维. 然后这八组QKV自己计算自己的attension, 计算完合成一个attension(重回512维), 然后再处理一下得到结果.

- 输入：token 的 512 维 embedding 向量
- 全局投影：分别乘`W_Q/W_K/W_V`，得到 3 个 512 维的总 Q、总 K、总 V 向量
- 分头：把总 Q/K/V 各拆成 8 个 64 维的子向量，每个头拿到专属的`Q_i/K_i/V_i`
- 分头计算：每个头独立执行缩放点积注意力，输出 8 个 64 维的上下文特征向量
- 拼接融合：把 8 个 64 维向量拼回 1 个 512 维的总向量
- 最终投影：乘`W_o`矩阵，得到最终的 512 维输出向量

## 残差连接, 归一化

将原token的向量Add到新向量上, 以此保持原token的特征. (Add)

然后将得到的新向量归一化. (norm)

## 输出

现在我们手里有了完美的隐藏状态向量，接下来就是把它变成人类能读的文字。这里要分两种最主流的 Transformer 架构来讲，但核心逻辑完全一致：

### Decoder-only 架构（比如 GPT、LLaMA，现在最常用的文本生成大模型）

这是最简单、最直观的生成流程，我们以 “续写句子：我今天去公园，\_\_\_\_” 为例：

核心逻辑：自回归生成（Autoregressive Generation）

> 每次只生成**一个**token，把这个 token 加回输入序列的末尾，再用新的序列生成下一个 token，循环往复，直到生成「结束符（\<EOS\>）」或达到最大长度。

具体步骤（对应上面的例子）：

1. ​**输入初始序列**：把 “我今天去公园，” 转成 token ID，输入模型，经过多头注意力 + FFN，得到每个 token 的最终隐藏状态。
2. ​**取最后一个 token 的隐藏状态**​：我们只关心序列**最后一个位置**的隐藏状态向量（形状`[d_model]`），因为它包含了前面所有上下文的信息，是预测下一个词的关键。
3. **线性层 + Softmax：把向量转成词汇表概率**

   - 线性层（LM Head）：把`d_model`​维的隐藏状态，映射到**词汇表大小**的维度（比如词汇表有 3 万个词，就映射到 3 万维）；
   - Softmax：把这 3 万维的数值，转成​**和为 1 的概率分布**（比如 “玩” 的概率是 0.4，“散步” 的概率是 0.3，“吃饭” 的概率是 0.001...）。
4. **采样：从概率分布里选一个 token**  
   不是直接选概率最大的（贪婪搜索），而是用**Top-k、Top-p（核采样）**  等策略，从概率高的 token 里随机选一个（比如选到了 “玩”）。

   1. ​**自回归循环**：把 “玩” 加回输入序列，现在输入变成了 “我今天去公园，玩”，重复步骤 1-4，生成下一个 token（比如 “耍”），直到生成结束符。
   2. **最终输出**：把生成的 token ID 串起来，转回文字，得到 “我今天去公园，玩耍了一整天，非常开心。”

原教程是这样讲的, 感觉更简化, 好理解:

> 无论是回答问题、写故事还是生成代码，本质上都是在一个已有的文本序列后面，一个词一个词地添加最合理的内容。基于这个思想，GPT 做了一个大胆的简化：**它完全抛弃了编码器，只保留了解码器部分。**  这就是 **Decoder-Only** 架构的由来。
>
> Decoder-Only 架构的工作模式被称为**自回归 (Autoregressive)**  。这个听起来很专业的术语，其实描述了一个非常简单的过程：
>
> 1. 给模型一个起始文本（例如 “Datawhale Agent is”）。
> 2. 模型预测出下一个最有可能的词（例如 “a”）。
> 3. 模型将自己刚刚生成的词 “a” 添加到输入文本的末尾，形成新的输入（“Datawhale Agent is a”）。
> 4. 模型基于这个新输入，再次预测下一个词（例如 “powerful”）。
> 5. 不断重复这个过程，直到生成完整的句子或达到停止条件。

---

### Encoder-Decoder 架构（比如机器翻译、文本摘要，Transformer 原文的经典结构）

它比 Decoder-only 多了一个**交叉注意力（Cross-Attention）**  步骤，但生成逻辑依然是自回归：

1. **Encoder 处理源语言**：经过多头注意力 + FFN，得到**源语言的上下文向量序列**。
2. ​**Decoder 自回归生成目标语言**：

   - 初始输入：给 Decoder 一个特殊的「起始符（\<SOS\>）」；
   - 以第一个文字作为Q, encoder的输出作为KV算attension, 然后同Decoder-only架构采样生成第一个字.
   - 然后将第一个字也纳入考量, 生成第二个字的时候, 将第二个字作为Q, 第一个字作为KV, 算attension. (这叫掩码多头注意力, 通过已有内容生成未知内容, 相当于多头注意力遮住后半部分)
   - 不仅要考量第一个字, 也同样要考虑一开始encoder输出的内容. 将第二个字作为Q, encoder输出的内容作为KV,  算attension. (这叫交叉注意力, 作用是“让目标语言的生成，时刻对齐源语言的语义”)
   - 上面提到的计算attension之后都要做Add&Norm操作, 最后要做FFN和Add&Norm操作. 整个流程构成解码器  
     ​![image](assets/image-20260310211223-vy1ochr.png)
   - 后续步骤和 Decoder-only 完全一致：FFN → 取最后一个隐藏状态 → 线性层 + Softmax → 采样生成第一个 token（比如 “I”）；
   - 循环：把 “I” 加回 Decoder 输入，生成下一个 token（“love”），直到生成结束符。
3. **最终输出**：把生成的 token 串起来，得到 “I love”。

让AI模仿原教程的口吻写的:

> Encoder-Decoder 架构也常被称为**序列到序列（Seq2Seq）**  架构。这个听起来很专业的术语，其实描述了一个非常简单的 “先彻底读懂源内容、再对照着逐词生成目标内容” 的过程，我们以把中文句子翻译成英文为例，完整过程是这样的：
>
> 1. 给架构里的「Encoder 编码器」输入完整的源文本，也就是我们要翻译的原句：“Datawhale 开源社区很棒”。
> 2. Encoder 会从头到尾读完、吃透这句话，通过注意力机制搞懂每个词的意思、词和词之间的关联，最终输出一份​**包含原句全部语义信息的语义笔记**，相当于给原句做了一份精准、全面的内容备忘录。
> 3. 把这份完整的语义笔记全程交给「Decoder 解码器」，同时给 Decoder 一个专门的「起始信号」，告诉它可以开始生成目标文本了。
> 4. Decoder 基于这个起始信号，一边参考自己已经生成的内容（最开始只有起始信号），一边对照着手里的语义笔记，预测出第一个最有可能的英文词：“Datawhale”。
> 5. Decoder 将自己刚刚生成的词 “Datawhale” 添加到自身输入的末尾，形成新的输入，同时始终拿着完整的语义笔记，再次预测下一个最有可能的词：“open”。
> 6. 不断重复这个过程：每生成一个新词，就把它追加到自己的输入里，每一步预测都对照着源文本的语义笔记，确保生成的内容和原句意思对齐，再预测下一个词。
> 7. 直到 Decoder 生成了专门的「结束信号」，或是达到了设定的最大生成长度，整个过程停止，最终我们就得到了和原句语义完全匹配的完整英文句子：“Datawhale open source community is great”。

## 关键补充：为什么生成的语言是 “自然” 的？（采样策略的作用）

如果每次都选概率最大的 token（贪婪搜索），生成的句子会非常生硬、重复，甚至不通顺。让语言变自然的核心，是​**采样策略**：

1. ​**Top-k 采样**​：只从概率最高的 k 个 token 里随机选（比如 k\=50），既保证了语义连贯，又引入了随机性；
2. ​**Top-p（核采样）** ​：从概率加起来刚好等于 p 的最小 token 集合里随机选（比如 p\=0.9），比 Top-k 更灵活，能自适应调整候选池大小；
3. ​**温度系数（Temperature）** ：调整 Softmax 的 “尖锐程度”—— 温度越高，概率分布越平坦，生成越随机；温度越低，分布越尖锐，生成越保守。

现在的大模型（比如 GPT-4o、Claude 3），都是用这些采样策略的组合，在 “语义准确” 和 “语言自然” 之间找到平衡。

在文本生成中，当同时设置 Top-p、Top-k 和温度系数时，这些参数会按照分层过滤的方式协同工作，其优先级顺序为：温度调整→Top-k→Top-p。温度调整整体分布的陡峭程度，Top-k 会先保留概率最高的 k 个候选，然后 Top-p 会从 Top-k 的结果中选取累积概率≥p 的最小集合作为最终的候选集。不过，通常 Top-k 和 Top-p 二选一即可，若同时设置，实际候选集为两者的交集。

# 分词

分词器按照一定**算法**, 用**语料库**(原始文字), 构建**词表**. 这一步是在训练模型的时候完成的.

比如用包含`low low low newer wider wider`的语料训练 BPE 分词器：

- 初始词表：`l, o, w, n, e, r, i, d, 空格` 等单个字符
- 迭代合并：先合并高频共现的`l+o→lo`​，再合并`lo+w→low`​，后续逐步合并`e+r→er`​、`w+i+d→wid`等
- 最终词表：会包含`low、new、er、wid、wider`等 token
- 推理使用：输入`lower`​，分词器会直接切分为`["low", "er"]`，完美处理训练语料中没出现过的新词，这也是子词分词器的核心优势。

日常使用的时候: 文本-\>分词器切分成token-\>将tokne处理成向量, 开始走transformer.

**分词器对开发者的意义**

理解分词算法的细节并非目的，但作为智能体的开发者，理解分词器的实际影响十分重要，这直接关系到智能体的性能、成本和稳定性：

- ​**上下文窗口限制**​：模型的上下文窗口（如 8K, 128K）是以 **Token 数量**计算的，而不是字符数或单词数。同样一段话，在不同语言（如中英文）或不同分词器下，Token 数量可能相差巨大。精确管理输入长度、避免超出上下文限制是构建长时记忆智能体的基础。
- ​**API 成本**：大多数模型 API 都是按 Token 数量计费的。了解你的文本会被如何分词，是预估和控制智能体运行成本的关键一步。
- ​**模型表现的异常**​：有时模型的奇怪表现根源在于分词。例如，模型可能很擅长计算 `2 + 2`​，但对于 `2+2`（没有空格）就可能出错，因为后者可能被分词器视为一个独立的、不常见的词元。同样，一个词因为首字母大小写不同，也可能被切分成完全不同的 Token 序列，从而影响模型的理解。在设计提示词和解析模型输出时，考虑到这些“陷阱”有助于提升智能体的鲁棒性。

在 `transformers`​ 库中，我们通常使用 `AutoModelForCausalLM`​ 和 `AutoTokenizer` 这两个类来自动加载与模型匹配的权重和分词器。下面这段代码会自动从 Hugging Face Hub 下载所需的模型文件和分词器配置，这可能需要一些时间，具体取决于你的网络速度。

```Python
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

# 指定模型ID
model_id = "Qwen/Qwen1.5-0.5B-Chat"

# 设置设备，优先使用GPU
device = "cuda" if torch.cuda.is_available() else "cpu"
print(f"Using device: {device}")

# 加载分词器
tokenizer = AutoTokenizer.from_pretrained(model_id)

# 加载模型，并将其移动到指定设备
model = AutoModelForCausalLM.from_pretrained(model_id).to(device)
```

# **模型选型的关键考量**

在为您的智能体选择大语言模型时，可以从以下几个维度进行综合评估：

- ​**性能与能力**：这是最核心的考量。不同的模型擅长的任务不同，有的长于逻辑推理和代码生成，有的则在创意写作或多语言翻译上更胜一筹。您可以参考一些公开的基准测试排行榜（如 LMSys Chatbot Arena Leaderboard）来评估模型的综合能力。
- ​**成本**：对于闭源模型，成本主要体现在 API 调用费用，通常按 Token 数量计费。对于开源模型，成本则体现在本地部署所需的硬件（GPU、内存）和运维上。需要根据应用的预期使用量和预算做出选择。
- ​**速度（延迟）** ：对于需要实时交互的智能体（如客服、游戏 NPC），模型的响应速度至关重要。一些轻量级或经过优化的模型（如 GPT-3.5 Turbo, Claude 3.5 Sonnet）在延迟上表现更优。
- ​**上下文窗口**：模型能一次性处理的 Token 数量上限。对于需要理解长文档、分析代码库或维持长期对话记忆的智能体，选择一个拥有较大上下文窗口（如 128K Token 或更高）的模型是必要的。
- ​**部署方式**：使用 API 的方式最简单便捷，但数据需要发送给第三方，且受限于服务商的条款。本地部署则能确保数据隐私和最高程度的自主可控，但对技术和硬件要求更高。
- ​**生态与工具链**：一个模型的流行程度也决定了其周边生态的成熟度。主流模型通常拥有更丰富的社区支持、教程、预训练模型、微调工具和兼容的开发框架（如 LangChain, LlamaIndex, Hugging Face Transformers），这能极大地加速开发进程，降低开发难度。选择一个拥有活跃社区和完善工具链的模型，可以在遇到问题时更容易找到解决方案和资源。
- ​**可微调性与定制化**：对于需要处理特定领域数据或执行特定任务的智能体，模型的微调能力至关重要。一些模型提供了便捷的微调接口和工具，允许开发者使用自己的数据集对模型进行定制化训练，从而显著提升模型在特定场景下的性能和准确性。开源模型在这方面通常提供更大的灵活性。
- ​**安全性与伦理**：随着大语言模型的广泛应用，其潜在的安全风险和伦理问题也日益凸显。选择模型时，需要考虑其在偏见、毒性、幻觉等方面的表现，以及服务商或开源社区在模型安全和负责任AI方面的投入。对于面向公众或涉及敏感信息的应用，模型的安全性和伦理合规性是不可忽视的考量。
